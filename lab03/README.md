# Лабораторная работа №3
## Поведенческие паттерны. Цепочка обязанностей (Chain of Responsibility)

---

## 1. Цель работы

Изучить и реализовать поведенческий паттерн проектирования **«Цепочка обязанностей»** (Chain of Responsibility) на примере системы многоуровневой технической поддержки с использованием LLM-моделей.

---

## 2. Теоретическая часть

**Цепочка обязанностей** — поведенческий паттерн, позволяющий передавать запросы последовательно по цепочке обработчиков. Каждый обработчик самостоятельно решает, обработать запрос или передать его дальше по цепочке.

### Ключевые участники паттерна:

| Участник | Роль |
|---|---|
| `Handler` (абстрактный) | Определяет интерфейс обработки и хранит ссылку на следующего обработчика |
| `ConcreteHandler` | Конкретный обработчик, реализующий логику обработки или делегирования |
| `Client` | Инициирует запрос, направляя его первому обработчику в цепочке |

---

## 3. Описание реализации

### 3.1 Общая архитектура

Система реализована в виде REST API (FastAPI) и моделирует службу технической поддержки с тремя уровнями эскалации. Каждый уровень использует отдельную LLM-модель.

```
Входящий тикет → BotHandler (GigaChat) → OperatorHandler (DeepSeek) → LeadHandler (OpenAI)
```

Если обработчик не знает ответа (ответ содержит `"НЕ ЗНАЮ"`), запрос передаётся следующему по цепочке.

На диаграмме представлены следующие связи:

- **`SupportHandler`** — абстрактный базовый класс с методами `handle()` и `set_next()`. Содержит **самоссылку** (`_next`) — ключевой элемент паттерна, хранящий ссылку на следующего обработчика.
- **`BotHandler`**, **`OperatorHandler`**, **`LeadHandler`** — конкретные обработчики, наследующие `SupportHandler`.
- **`SupportDepartment`** — вспомогательный класс-строитель, управляющий сборкой цепочки через метод `add_specialist()`.
- **`Ticket`** — класс-контейнер, который служит «паспортом» заявки. Он «путешествует» через всю цепочку обработчиков, накапливая в себе историю прохождения (history), ответы нейросетей (llm_logs) и метаданные (контент и уровень сложности), позволяя передавать состояние от одного специалиста к другому.

---

### 3.3 Описание классов

#### Класс `Ticket`
Объект передаваемого запроса (аналог «письма» в цепочке). Хранит:
- `content` — текст обращения;
- `level` — приоритет тикета;
- `history` — список обработчиков, которые касались тикета;
- `llm_logs` — логи ответов каждой модели.

#### Абстрактный класс `SupportHandler`
Реализует базовую логику паттерна:
```python
class SupportHandler(ABC):
    def __init__(self, next_handler=None):
        self._next = next_handler       # Ссылка на следующий обработчик

    def set_next(self, h):
        self._next = h
        return h                        # Fluent interface для цепочечных вызовов

    @abstractmethod
    def handle(self, ticket: Ticket) -> str:
        if self._next:
            return self._next.handle(ticket)   # Передача по цепочке
        return "Тикет закрыт: ни один специалист не смог помочь."
```

#### Класс `BotHandler` — 1-я линия (GigaChat)
- Обрабатывает простые случаи напрямую (ключевые слова `"пароль"`, `"оплата"`).
- В остальных случаях обращается к GigaChat.
- При ответе `"НЕ ЗНАЮ"` — передаёт тикет оператору.

#### Класс `OperatorHandler` — 2-я линия (DeepSeek)
- Вызывает модель DeepSeek.
- При `"НЕ ЗНАЮ"` — передаёт тикет техлиду.

#### Класс `LeadHandler` — 3-я линия (OpenAI)
- Финальный обработчик в цепочке.
- Использует OpenAI.
- Если и он не знает — цепочка завершается базовым сообщением родительского класса.

#### Класс `SupportDepartment`
Вспомогательный класс для динамического построения цепочки:
```python
department.add_specialist(BotHandler()) \
          .add_specialist(OperatorHandler()) \
          .add_specialist(LeadHandler())
```
Использует паттерн **Fluent Interface** — каждый вызов `add_specialist()` возвращает `self`.

### 3.4 Логика обработки запроса

```
POST /handle
    │
    ▼
Создаётся Ticket(content, level)
    │
    ▼
BotHandler.handle(ticket)
    ├── "пароль" / "оплата" → немедленный ответ
    ├── Бот знает → вернуть ответ
    └── "НЕ ЗНАЮ" → OperatorHandler.handle(ticket)
                        ├── DeepSeek знает → вернуть ответ
                        └── "НЕ ЗНАЮ" → LeadHandler.handle(ticket)
                                            ├── OpenAI знает → вернуть ответ
                                            └── "НЕ ЗНАЮ" → "Тикет закрыт"
```

## 4. API-интерфейс

### `POST /handle`
**Запрос:**
```json
{
  "content": "Текст обращения пользователя",
  "level": 1
}
```

**Ответ:**
```json
{
  "result": "Ответ на обращение",
  "path": ["Бот-автоответчик (GigaChat)", "Оператор 2-й линии (DeepSeek)"],
  "resolved_by": "op",
  "llm_logs": {
    "bot": "НЕ ЗНАЮ",
    "op": "Ответ оператора"
  }
}
```

### `GET /health`
Возвращает статус сервиса и состояние API-ключей для всех трёх LLM.

## 5. Соответствие паттерну

| Элемент паттерна | Реализация в коде |
|---|---|
| Абстрактный обработчик | `SupportHandler` (ABC) |
| Конкретные обработчики | `BotHandler`, `OperatorHandler`, `LeadHandler` |
| Ссылка на следующий | `self._next` в `SupportHandler` |
| Передача по цепочке | `super().handle(ticket)` при `"НЕ ЗНАЮ"` |
| Сборка цепочки | `SupportDepartment.add_specialist()` |
| Объект запроса | `Ticket` с историей и логами |

## Вывод

В ходе лабораторной работы был реализован паттерн **«Цепочка обязанностей»** на примере многоуровневой LLM-системы поддержки. Паттерн позволил:

1. Разделить ответственность между тремя обработчиками (бот, оператор, техлид).
2. Обеспечить автоматическую эскалацию тикета при невозможности дать ответ.
3. Сохранить низкую связанность: каждый обработчик не знает о существовании других, кроме непосредственно следующего.

Реализация соответствует классическому описанию паттерна GoF и демонстрирует его применимость в современных AI-системах.
