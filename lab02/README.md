# Лабораторная работа №2: структурные паттерны проектироания
## Паттерн проектирования «Фасад»

## Цель работы

Изучить и реализовать структурный паттерн проектирования **Фасад (Facade)** на практическом примере — симуляции кофемашины с HTTP-сервером.

---

## Теоретическая часть

### Что такое паттерн Фасад?

**Фасад (Facade)** — структурный паттерн проектирования, который предоставляет простой интерфейс к сложной системе классов, библиотеке или фреймворку.

### Проблема, которую решает паттерн

Когда программа работает со множеством объектов сложной подсистемы, код становится тесно связанным с деталями реализации. Клиентский код вынужден знать о существовании всех внутренних компонентов и управлять ими напрямую.

### Решение

Фасад — это класс, предоставляющий простой интерфейс для работы со сложной подсистемой. Он:
- скрывает внутреннюю сложность системы
- делегирует запросы клиента нужным объектам подсистемы
- управляет их жизненным циклом

### Структура паттерна

```
┌─────────────┐         ┌──────────────────────┐
│   Client    │────────▶│       Facade         │
└─────────────┘         │  + operation_A()     │
                        │  + operation_B()     │
                        └──────────┬───────────┘
                                   │ использует
              ┌────────────────────┼─────────────────────┐
              ▼                    ▼                      ▼
     ┌──────────────┐   ┌──────────────────┐   ┌──────────────────┐
     │ SubsystemA   │   │  SubsystemB      │   │  SubsystemC      │
     └──────────────┘   └──────────────────┘   └──────────────────┘
```

### Когда применять

- Когда нужен простой или урезанный интерфейс к сложной подсистеме.
- Когда необходимо разложить подсистему на отдельные слои.
- Когда нужно снизить связанность между клиентским кодом и деталями реализации.

## Практическая часть

### Описание задачи

Реализована система управления кофемашиной **CafeOS** — приложение на C++ с HTTP REST API. Система моделирует работу реальной кофемашины со всеми её механическими подсистемами.

### Архитектура проекта

```
CafeServer  (HTTP-сервер, точка входа)
    │
    └──▶ CoffeeMachineFacade  ◀── ФАСАД
              │
              ├──▶ Storage          (хранилище ингредиентов)
              ├──▶ Grinder          (кофемолка)
              ├──▶ Brewer           (заварочный узел)
              ├──▶ MilkFrother      (капучинатор)
              ├──▶ WaterHeater      (бойлер)
              └──▶ SugarDispenser   (дозатор сахара)
```

### Компоненты системы

#### Подсистемы (Subsystems)

| Класс | Роль | Метод |
|-------|------|-------|
| `Storage` | Хранилище ингредиентов | `check_ingredients()`, `consume_*()`, `refill_*()` |
| `Grinder` | Кофемолка | `grind(amount_g, storage)` |
| `Brewer` | Заварочный узел | `brew_espresso(water_ml, storage)` |
| `MilkFrother` | Капучинатор | `froth_milk(amount_ml, storage)` |
| `WaterHeater` | Бойлер | `dispense_hot_water(amount_ml, storage)` |
| `SugarDispenser` | Дозатор сахара | `add_sugar(amount_g, storage)` |

#### Фасад — `CoffeeMachineFacade`

Скрывает взаимодействие всех подсистем за простым интерфейсом:

```cpp
class CoffeeMachineFacade {
public:
    void make_espresso();    // 15г зерен + 30мл воды
    void make_americano();   // 15г зерен + 30мл + 90мл воды
    void make_cappuccino();  // 15г зерен + 30мл воды + 100мл молока
    void make_latte();       // 15г зерен + 30мл воды + 150мл молока
    void add_sugar(int portions);
    void refill_storage(const string& ingredient);
    int  get_ingredient(const string& ingredient) const;
};
```

**Без фасада** клиент вынужден был бы вручную:
1. Проверить наличие ингредиентов в `Storage`
2. Вызвать `Grinder::grind()`
3. Вызвать `Brewer::brew_espresso()`
4. Вызвать `MilkFrother::froth_milk()`
5. Обработать все ошибки

**С фасадом** — один вызов `make_cappuccino()`.

#### Сервер — `CafeServer`

Является клиентом фасада. Предоставляет REST API:

| Метод | Endpoint | Описание |
|-------|----------|----------|
| `GET` | `/` | Главная страница (index.html) |
| `GET` | `/storage` | Текущий уровень запасов (JSON) |
| `POST` | `/brew` | Приготовить напиток |
| `POST` | `/refill` | Пополнить запасы |

### Рецепты напитков

| Напиток | Зерна | Вода (эспрессо) | Доп. вода | Молоко |
|---------|-------|-----------------|-----------|--------|
| Эспрессо | 15 г | 30 мл | — | — |
| Американо | 15 г | 30 мл | 90 мл | — |
| Капучино | 15 г | 30 мл | — | 100 мл |
| Латте | 15 г | 30 мл | — | 150 мл |

### Начальные запасы (`Storage`)

| Ингредиент | Начальный объём | После пополнения |
|------------|-----------------|------------------|
| Вода | 1000 мл | 1000 мл |
| Кофейные зерна | 200 г | 200 г |
| Молоко | 500 мл | 500 мл |
| Сахар | 100 г | 100 г |

## Сборка и запуск

### Зависимости

- Компилятор C++17 (g++ / clang++)
- Библиотека [cpp-httplib](https://github.com/yhirose/cpp-httplib) (header-only, файл `httplib.h`)

### Компиляция

```bash
g++ -std=c++17 -o cafeos main.cpp -lpthread
```

### Запуск

```bash
./cafeos
# CafeOS Server (All-in-One) запущен на http://localhost:8080
```

### Примеры запросов (curl)

```bash
# Получить текущие запасы
curl http://localhost:8080/storage

# Приготовить капучино с 2 ложками сахара
curl -X POST http://localhost:8080/brew \
     -H "Content-Type: application/json" \
     -d '{"type": "cappuccino", "sugar": 2}'

# Приготовить американо
curl -X POST http://localhost:8080/brew \
     -H "Content-Type: application/json" \
     -d '{"type": "americano", "sugar": 0}'

# Пополнить молоко
curl -X POST http://localhost:8080/refill \
     -H "Content-Type: application/json" \
     -d '{"ingredient": "milk"}'

# Пополнить всё
curl -X POST http://localhost:8080/refill \
     -H "Content-Type: application/json" \
     -d '{"ingredient": "all"}'
```

### Пример ответа `/storage`

```json
{
  "water": 940,
  "beans": 185,
  "milk": 400,
  "sugar": 90
}
```

## Преимущества паттерна (вывод)

| Критерий | Без фасада | С фасадом |
|----------|-----------|-----------|
| Сложность клиентского кода | Высокая | Низкая |
| Связанность (coupling) | Высокая | Низкая |
| Читаемость | Трудно | Легко |
| Тестируемость | Сложно | Просто |
| Замена подсистем | Ломает клиент | Прозрачна |

**Паттерн Фасад** позволил:
1. Изолировать 5 независимых подсистем за единым интерфейсом
2. Упростить `CafeServer` — он работает только с `CoffeeMachineFacade`, не зная о `Grinder`, `Brewer` и прочих
3. Легко расширять систему — добавление нового напитка требует изменений только в фасаде
